# 第三章 系统需求分析

本章从可行性、角色与用例、功能性与非功能性四个方面，对「基于 Spring Boot 与 Vue 的智能团队任务管理与绩效评估系统」进行需求分析。功能需求条目的划分与**第四章详细设计**、**第五章系统实现**保持模块同构，便于论文前后对照。本章涉及可视化的图形说明（如 §3.2.2 用例总览）采用与第四章相同的体例：**一、图式类型（须先确定）**；**二、该类图的基本规范（须严格遵守）**；**三、绘制提示词（成图自检清单）**。

## 3.1 可行性分析

### 3.1.1 技术可行性

系统采用成熟的主流技术栈：后端 Spring Boot 3.x、Spring Security、MyBatis 与 MySQL；前端 Vue 3、TypeScript、Vite 与 Ant Design Vue。上述技术文档与社区资源丰富，开发与部署风险可控。绩效与指派场景中涉及的层次分析法（AHP）、模糊综合评价（FCE）、指数平滑与风险概率模型均可在服务端以纯 Java 实现，无需依赖昂贵商业中间件，技术路线可行。

### 3.1.2 经济可行性

开发以开源框架为主，无额外运行时授权费用；硬件上普通 PC 或云主机即可支撑中小团队并发。相较采购重型项目管理套件，本系统适合毕业设计或校内、小型团队试点，经济成本可接受。

### 3.1.3 操作可行性

界面为中文 Web 管理端，角色权限清晰：系统管理员负责平台与用户；团队管理者负责任务、绩效与 AHP 配置；普通成员完成任务与填报工时。浏览器即可访问，学习成本低于安装客户端的专业工具，具备操作可行性。

## 3.2 用户角色与用例分析

### 3.2.1 角色定义

系统实现三类角色，与 Spring Security 中角色编码一致：


| 角色    | 编码      | 主要职责                                       |
| ----- | ------- | ------------------------------------------ |
| 系统管理员 | ADMIN   | 全平台用户管理、审计查询、系统参数（FCE 权重、负荷平滑系数等）          |
| 团队管理者 | MANAGER | 所辖团队的任务、成员审批、AHP 权重、绩效周期、风险与指派、技能可选项、管理者看板 |
| 普通成员  | MEMBER  | 参与团队任务、填报工时、互评、查看我的绩效与消息、维护个人信息            |

## 3.3 功能性需求分析

以下编号与**第四章、第五章**同序小节对应，便于设计、实现与评审对照；每条只写「系统应做到什么」，细节放在后文。

### 3.3.1 用户认证与会话（§4.2.1，§5.2）

用户以账号口令登录；服务端签发 JWT 访问令牌与刷新令牌，除登录、刷新等公开接口外，业务请求须携带合法访问令牌。访问令牌过期时，客户端宜静默刷新并重试；刷新失败则清空本地凭据并回到登录页，前端对受保护路由做同步拦截。

### 3.3.2 全局布局与团队上下文（§4.2.2～4.2.3，§5.3）

登录后主布局按角色展示菜单、面包屑、消息入口与退出。凡依赖「当前团队」的任务、AHP、绩效等入口，须先选定团队并绑定 `teamId` 再进入；管理端路径与 `/api/admin/`** 仅管理员可达。

### 3.3.3 系统管理员：用户管理（§4.3.1，§5.4）

管理员可查询、创建、更新用户，可重置密码、启停账号；响应中不得返回密码哈希。

### 3.3.4 系统管理员：审计日志（§4.3.2，§5.5）

管理员可按时间倒查操作留痕，支持筛选与单次查询条数上限，覆盖任务、绩效、系统配置等关键事件。

### 3.3.5 系统管理员：系统参数（§4.3.3，§5.6）

全局运行参数仅管理员可读、可写，宜支持一次提交多项（至少含模糊综合评价三项权重与负荷平滑系数）；读出的应是库值与默认合并后的现行口径，保存须校验合法性并写入审计，不落敏感字段。

### 3.3.6 团队与成员（§4.4，§5.7）

团队管理者维护团队信息与成员入团审批；成员可查看本人归属及审批状态。任务、绩效、AHP 等数据均以团队为边界隔离。

### 3.3.7 团队任务协作（§4.5，§5.8）

在同一团队下，须支持任务的创建、查询、更新与改派，并用版本或乐观锁避免互相覆盖；可配置前置依赖与状态推进规则。成员改状态应走申请—审批，结果通知相关人。进度、截止等变化应触发风险重算，跨阈值时提醒团队管理者，并支持团队级风险刷新。团队管理者可按任务所需技能获取推荐人选列表，综合技能匹配、负荷与历史绩效及团队 AHP 权重排序，供人工确认后指派。

### 3.3.8 工时与负荷预测（§4.6，§5.9）

成员按自然周填报工时，系统持久化并支持本人及授权管理者查看未来若干周的负荷预测；平滑系数取自全局配置。

### 3.3.9 AHP 权重（§4.7，§5.10）

团队管理者维护三准则互反矩阵，一致性未通过则禁止保存；通过后权向量参与指派推荐，未配置时使用默认均分。

### 3.3.10 绩效周期与同事互评（§4.8.1、4.8.2，§5.11）

团队管理者可开启、关闭绩效周期；周期内支持同事按维度互评，禁止自评与对同一对象重复提交。关账时并入上级评价与系统客观维度，按全局权重做模糊综合评价，生成每人周期报告并更新历史绩效口径。

### 3.3.11「我的绩效」与趋势（§4.8.3，§5.12）

成员以只读方式查看本人各已关闭周期的得分及历史走势，与团队侧绩效管理页面职责分开。

### 3.3.12 个人信息与档案审批（§4.9，§5.13）

成员维护个人信息；涉审批字段在待审期间应有明确状态提示，管理员处理与消息提醒衔接。

### 3.3.13 消息中心（§4.10，§5.14）

系统提供统一收件箱，汇聚风险预警、任务审批、档案审批等；顶栏展示未读数，支持已读标记与跳转关联业务。

### 3.3.14 技能可选项（§4.11 技能，§5.15）

全局技能字典供查阅；团队管理者维护本团队可选技能子集，并与任务、成员技能及推荐接口保持同一编码空间。

### 3.3.15 管理者看板（§4.11 看板，§5.16）

团队管理者或管理员在选定团队后查看任务、风险、绩效或工时等聚合图表；普通成员访问看板入口时应导向与其角色相符的页面（如个人任务）。

## 3.4 非功能性需求分析

以下从**性能、安全、体验与可靠**几方面约定系统应满足的底线，与**第六章**测试维度大致对齐；具体阈值以部署环境与实测为准，本章不强行写死数字。

### 3.4.1 性能

在中小团队常见并发下，任务列表、看板聚合、用户与消息等核心接口应保持可接受的响应与分页体验；若论文或验收需要量化指标，可在测试环境测定 P95 等并写入第六章，此处只要求「明显卡顿不可接受」。

### 3.4.2 安全性

密码须单向哈希存储，敏感传输依赖部署侧 **HTTPS**；业务接口统一采用 **JWT** 与角色授权，管理端路径与普通业务隔离。还需防范**水平越权**：即便已登录，也不应通过篡改团队编号等方式读取他人团队数据，团队管理者与成员的数据范围由规则在服务层收口。

### 3.4.3 可用性与可维护性

界面与交互宜风格统一，关键操作有明确成功或失败反馈；运行参数集中管理，重要配置变更应能留痕审计，降低后期运维与版本升级时的沟通成本。

### 3.4.4 可靠性

任务创建与状态变更、绩效关账等关键路径应在事务边界内完成，避免出现半截成功。多人同时改同一任务等场景下，须有冲突检测与可读提示，禁止静默覆盖他人修改。

## 本章小结

本章完成了可行性论证，将用户划分为管理员、团队管理者、成员三类并与实现对齐；按与第四、五章同构的十五条功能需求梳理了认证布局、系统管理、团队任务、工时与预测、AHP、绩效与互评、个人与消息、技能与看板等能力边界，并从性能、安全、可用与可靠等方面补充了非功能约束；**§3.2.2** 给出了与 **§3.3** 对位的**图 3-1** 用例图绘制体例。后续第四章将在本章需求基础上给出架构与详细设计，第五章给出具体实现与界面验证。

# 第四章 系统架构与详细设计

第三章已说明系统要具备哪些能力。本章只回答**实现侧怎么串起来**：一次请求从前端到落库经过哪些环节、状态与规则放在哪一层校验、读与写如何分流。叙述尽量不用源码里的类名与路径；数据库一节为便于与脚本对照，仍保留表名与字段表。各节末「**【图 4-x 绘制说明】**」供成图替换。

## 4.1 系统总体架构设计

浏览器里用单页应用组织页面与路由，所有业务访问走同源下的 **JSON 接口**；本地开发时由打包工具把接口前缀代理到后端，上线后通常交给网关或反向代理，这样前端不必写死后端地址。服务端采用**分层**：靠外的一层只负责协议、入参与返回视图；中间一层承载事务与「是否属于本团队、是否已审批」等业务断言；再往里是访问数据库的映射。容易随产品调整的形状（技能向量、判断矩阵、互评打分、绩效明细等）用可伸缩字段存放，由程序在写入前检查结构；带公式的评价与预测放在**独立的小计算模块**里，由中间层在关账、指派、预测等节点调用，避免把算式摊在接入层。

登录后发放**两种令牌**：短寿命的访问凭据与可换新的刷新凭据；安全过滤器在每条请求很早的阶段解析访问凭据并识别身份与角色，配置里把「仅管理员」的整段地址与「其余须登录」分开。但**身份不等于授权**：能否读写某个团队的数据，仍在业务入口用「是否团队负责人、成员是否已通过入团审批」等条件再判断一次，形成**先挡身份、再挡团队**两道闸。

## 4.2 功能模块、角色与路由结构设计

根据第三章对系统**功能性需求**的分析，本小节从用户可见功能边界出发，归纳 Wise-Team 的**基本功能模块**及其向下展开的**子功能**，其层次关系见**图 4-14**（功能结构图）。顶层为系统总称；第二层划分为六大关键模块，依次为：**用户认证与权限**、**系统管理**、**团队与成员**、**任务协作**、**工时与绩效**、**消息与看板**。第三层在各模块下列出若干子功能，例如登录与双令牌机制、用户与审计及全局参数与技能字典、团队维护与入团审批及团队 AHP、任务维护与依赖进度及状态审批与风险智能指派、周工时与负荷预测及绩效互评与关账、站内消息与档案审批及管理者数据看板等，与第三章需求分类及后续各节路由、接口叙述相呼应。（*若学院模板将该图置于 §4.2 之首且单独编号，可将「图 4-14」改为「图 4.2」并与原文中泳道图编号协调。*）

### 4.2.1 角色模型与权限边界

三类角色写进令牌并在配置里区分：管理端整段地址要求管理员角色，其余在「已登录」前提下进入业务。具体能否操作某个团队的数据，在业务入口比对「是否该团队负责人」「成员是否已通过入团」——这样**网络层挡身份、业务层挡团队**，日志里也易区分是未登录、无角色还是业务拒绝。

### 4.2.2 菜单与路由树

登录页与带侧栏的业务壳分开；壳内路由用元信息标记「须先登录」。侧栏按本地角色决定菜单长什么样，但**是否返回数据**只信服务端。凡路径里要带团队编号的，先弹出团队列表再跳转，避免手改地址误入他人团队。

### 4.2.3 团队上下文设计

团队列表对管理员与团队负责人返回不同范围：前者见全部，后者只见自己负责的那些。选定后把团队编号写进地址，面包屑用中文拼出「当前在哪个团队语境」。顶栏铃铛定时用轻量接口刷新未读数，与谁往收件表里插消息互不牵连。

## 4.3 系统管理子系统详细设计

用户侧：列表与编辑接口在返回前剥掉密码摘要，避免敏感面扩大。审计侧：关键动作在业务事务成功后再追加一行流水，防止「事没办成却留痕」。参数侧：读取时先把内置默认的几项权重和平滑系数放进内存，再用库里已存值覆盖，管理页因此总能看到完整一组；保存时整包写入键值表并记一条配置变更审计。绩效关账与负荷预测在运行期都走同一套「读参数」逻辑；若平滑系数被填坏，解析失败就退回配置文件里的保底小数，预测接口仍能返回结果。

**【图 4-4 绘制说明】**

**一、图式类型（须先确定）**  
**数据流图（DFD，Structured Analysis）**，采用 **Gane–Sarson** 或 **Yourdon／DeMarco** 符号体系中的一种，**全文统一**；本图为零层或按子系统**物理分区**的 DFD，表达系统管理中的数据从外部到加工再到存储的流动。

2. **处理（加工）**：**圆角矩形**（Gane–Sarson）或圆泡（Yourdon）；写**动词短语**，并编号 P1、P2…。
3. **数据存储**：**开口矩形**或双横线端；标注 **D1** 等编号及存储名（用户表、审计库、配置键值表等）。
4. **数据流**：**有向箭头**，旁写**数据名或数据组名**；禁止无标签箭头。
5. **平衡**：每个加工须有输入、输出流；若简化直连须在图注声明。

**三、绘制提示词（成图自检清单）**  

- 含「用户 CRUD、审计查询、系统参数读／写」三组加工或三列等价布局。

## 4.4 团队与成员管理子系统详细设计

团队与成员是多对多：用一张中间表同时表示「想加入」和「已是成员」，用审批状态区分。此后凡团队内的写操作，流程都相似——先确认调用者与该团队的关系：要么你是负责人，要么你是已通过审批的成员；否则直接拒绝。这样「能不能协作」只在中间表上多一列状态，在业务入口用一套判定复用，不必每个模块各写一套字符串。

## 4.5 任务协作子系统详细设计

任务永远挂在某个团队之下。更新时带版本号：用「旧版本」去匹配更新，配不上就认为两人同时改冲突，从数据库一侧避免后写默默覆盖先写。前后依赖放在独立表里，改状态前先检查前置是否完成。成员要改状态，先写申请单，负责人批过才改任务主体，并往收件箱塞一条结果。进度或截止变化会触发风险重算：把进度、剩余时间、难度和执行人历史延误等折成延期概率和色区；若等级跨档上升，就给负责人发预警。推荐只做排序：在已通过成员里算技能贴合、负荷余量和历史表现，再按团队保存的三维权重加权；**返回名单但不自动改执行人**，由负责人在编辑任务时自己点选。

**一、图式类型（须先确定）**

## 4.6 工时采集与负荷预测子系统详细设计

成员按自然周填报实际工时：同一用户同一周再次提交，表现为覆盖更新而不是多行。预测时拉一段历史实际工时做指数平滑；平滑强度读全局参数，读坏了就用配置文件里的默认小数，接口仍能给出未来几周曲线和与已排任务结合的负荷感，页面不至于白屏。

## 4.7 AHP 权重子系统详细设计

团队管理者维护一张三准则互反矩阵：保存前先算出权向量和一致性比率，达不到常见阈值就整单拒绝，库里仍保留旧配置或空配置，避免「坏矩阵」污染后续指派。通过后把矩阵与权重序列化成可伸缩字段落库。智能指派在算分时并不直接翻矩阵，而是走「读当前团队权重」的窄口：若从未配过，就退回三等分默认权重，这样推荐列表在 AHP 空白时也能算。矩阵校验与落库收在一处，指派侧只认权向量，耦合面小。

**【图 4-8 绘制说明】**

**一、图式类型（须先确定）**  
**程序流程图（Program Flowchart）**，符号依据 **ISO 5807**／国内教材通行约定；一角附 **3×3 矩阵示意图** 作为**数据示意**（非流程图符号），须在图注说明「矩阵仅为示意，非 ISO 5807 符号」。

**二、该类图的基本规范（须严格遵守）**（流程图部分）  

1. **起止**：**圆角矩形**或**椭圆**表示开始／结束。
2. **处理**：**矩形**，内写处理说明。
3. **判断**：**菱形**，每一出口标注**条件**（是／否或表达式）。
4. **流线**：**有向箭头**；线须从出口连到唯一目标，避免悬空。
5. **禁止**：菱形无出口、无「否则」分支；判断与处理形状混用。

**（矩阵示意）** 行列标题写三准则名称，格内可写「互反比」占位数字，不追求与真实数据一致。

**三、绘制提示词（成图自检清单）**  

- 流程：输入矩阵 → 计算 CR → 判断「CR 是否小于 0.1」→ 否则返回调整、是则持久化矩阵与权向量。  
- 旁路：无配置时「三等分默认」指向指派加权。  
- 另支箭头：读取权向量 →「智能指派加权」框。  
- 矩阵行列头：技能匹配、负载、绩效。

## 4.8 绩效周期与多源评价子系统详细设计

团队侧：开周期、关周期、关账都走同一套绩效事务边界——开启时落周期行并记审计；周期内互评逐条入库，库级唯一约束挡住「同一人对同一被评人重复交卷」。关账时把管理者手填的上级评价、系统从任务汇总出的客观分、以及同伴各维度分聚起来，按全局参数里的三项权重归一后交给模糊综合评价模块算出综合分，写入周期个人报告表，并回写用户档案上的长期绩效口径，供后续指派里的「历史表现」维度读取。成员侧：看本人各周期得分与走势走只读查询路径，在入口强制「只能查自己，除非你是管理员或对该人有管理语境」，与关账写报告不是同一条调用链，避免误把读接口当写口用。

**【图 4-9 绘制说明】**

**一、图式类型（须先确定）**  
**数据流图（DFD，Gane–Sarson）**，以外部实体「团队管理者」「成员」与加工「系统／综合评价」等表达 §4.8 绩效周期与多源评价的数据流；**「我的绩效」只读查询**须为独立分支，数据流**不得**汇入关账写报告路径。

## 4.12 数据库设计

功能上要求数据能按团队隔离、关键动作可审计；非功能上要求主外键与唯一约束把「不该发生的事」挡在库里。本节用概念图归纳实体与联系，用逻辑表与版本化迁移脚本对齐字段与约束，核心七张表在下一小节逐列列出，其余表用文字概括意图，便于后文实现与测试共用同一套持久化真值。

### 4.12.1 概念结构设计

以下用实体—联系语言归纳边界，不复述需求条文。

**概念归纳。** **团队是协作的边界**，绝大多数业务都挂在某个团队下面；**用户既是登录主体，又是任务里的具体的人**——谁创建、谁执行、谁审批、谁收消息，都离不开用户这张「脸」。一个人可以加入多个团队，一个团队里有多名成员，因此「用户—团队」天然是多对多；中间那张成员关系表不仅拆开了多对多，还顺手承载了**入团要不要批、批没批**这类状态。每个团队再指向一名**负责人**，表示谁对这块地盘负责。

团队一旦立住，任务、绩效周期、团队级的 AHP 配置、互评记录等，都可以理解为**从团队伸出去的一束束一对多关系**；单条任务还会再指向**创建人**和**执行人**，把「谁提的需求、谁在做」钉在数据上。任务与任务之间若有先后关系，用依赖表表达有向的前后衔接；成员想改任务状态、要走审批，则单独用**状态变更申请**这张表把「申请—待审—通过/驳回」这条生命周期接住，和任务主表解耦，日后查日志也好说清。

绩效这边，一个周期里希望**每个成员最终有一份综合报告**，互评则是「谁评了谁、各维度多少分」的网状关系，业务上要求**同一人不能对同一被评人重复交卷**。工时按人、按周记一条事实；消息、档案变更申请从用户一侧一对多挂出去即可。全局参数和审计日志比较特别：前者就是键值对，后者记下谁在什么时候动了哪类资源——审计里操作者未必还建外键强绑用户表，免得人删了账号却连历史都查不成。技能方面，全局字典和团队可选用子集在实现里落在同一张「技能选项」表，用范围字段区分即可，概念上仍看作**一类实体、两种用途**，画图时不必硬拆成两个框。

**【图 4-13 绘制说明】**

下面这段是给「动手画图」用的约定，和课堂里讲的陈氏 E-R 一致，照着画不容易返工。

**一、图式类型**  
仍画**总览级、不带属性**的 **Peter Chen E-R 图**：矩形表示实体，菱形表示联系，连线上标清 **1** 与 **n**。图中**每一个实体矩形**仍应对应库中的一张物理表（从 **users** 到 **skill_options** 等，一张表一个框，勿遗漏），但**不在图中写列名、类型或键符号**——与业务主线绑得最紧的 **users、teams、team_members、tasks、performance_cycles、performance_reports、peer_reviews** 七张表，列级定义集中在 **§4.12.2** 的**表 4-1～表 4-7**；其余辅助表的用途与主要字段见同节**表 4-8**，成图时这些辅助矩形可略小、靠边，以突出「用户—团队—任务—绩效」主链。

**二、基本规范（成图时请逐项自检）**  
实体只用**矩形**写名称（中文职责或英文表名全文统一即可）；联系用**菱形**，里面写「属于」「依赖」「申请」这类短语；实体与菱形之间用**无向线**，基数标在**靠近实体的那一端**。多对多必须通过**联结实体**或规范的菱形拆法表达，例如本系统的用户—团队经**团队成员**表拆开，任务—任务的前置关系经**任务依赖**表拆开，不要直接画一条含糊的 n∶n 线。同一用户实体若被多处引用，宁可多画几条线分别标基数，也不要揉成一根「万能线」。图上不必写 SQL 级联、外键英文名或 `ON DELETE` 策略。

**三、布局与内容提示**  
团队矩形建议放在**中上或中心**，任务、绩效周期、互评等挂在团队一侧；**团队 AHP、工时、消息、审计**等亦须出现矩形以保持与实现一致，但不必与核心七表同大。用户经团队成员与团队相连；任务与用户之间用「创建」「执行」等联系体现，**不要写字段名**。**技能选项**只画一个矩形，用图例说明「全局行与团队行在同表」。图下写清图注：本图为无属性总览；**核心表字段及类型见表 4-1～表 4-7**；**九张支撑表及主要字段见表 4-8**；更细的列级定义以版本化迁移脚本为准；并简单解释矩形、菱形与基数记号。

### 4.12.2 逻辑结构设计

持久化要落实两件事：每张表只表达一种稳定事实，主键、外键与唯一约束把业务粒度钉死；易变形状用可伸缩列承载并由程序校验。工程侧用带版本号的迁移脚本演进库结构，脚本里的表名与约束名与运行库、与下文字段表应一致。主键多为自增长整型，时间戳记录创建与更新，能建外键的关联尽量建，具体级联策略以脚本为准。

## 5.2 用户认证与会话管理

**实现思路**  
身份认证在后端由 `AuthController` 对外提供登录与令牌刷新两类 REST 接口；`AuthService` 借助 Spring Security 的 `AuthenticationManager` 完成用户名与口令校验后，由 `JwtTokenProvider` 签发访问令牌与刷新令牌，从而在服务端维持无会话的凭据模型。每个业务请求进入时，`JwtAuthenticationFilter` 从 HTTP `Authorization` 首部解析 Bearer 访问令牌，在类型与签名校验通过后构造 `UserPrincipal` 并写入 `SecurityContext`，供后续控制器与服务层读取当前用户及其角色。前端在 `LoginView` 中将双令牌写入浏览器本地存储；全局 HTTP 客户端 `client.ts` 通过请求拦截器为出站请求自动附带访问令牌，并在收到 401 时尝试以刷新令牌静默换新后重发原请求，使用户在长时间操作过程中尽量不因令牌过期而被打断。

**核心代码片段**（节选自 `SecurityConfig.java`、`JwtAuthenticationFilter.java`、`client.ts`）

```java
// 无会话：不用服务器 Session，每条请求带 JWT 即可识别用户；关闭 CSRF 以适配前后端分离的 JSON 接口

    .sessionManagement(s -> s.sessionCreationPolicy(SessionCreationPolicy.STATELESS))
    .authorizeHttpRequests(auth -> auth
        .requestMatchers("/api/auth/**").permitAll()           // 登录、刷新放行
        .requestMatchers("/api/admin/**").hasRole("ADMIN")   // 管理端接口仅管理员
        .anyRequest().authenticated())                        // 其余须已登录
    .addFilterBefore(jwtAuthenticationFilter, UsernamePasswordAuthenticationFilter.class); // 在用户名密码过滤器之前解析 JWT
```

## 5.11 绩效周期与同事互评

**实现思路**

```java
// 读系统配置中三项权重并归一，保证三者之和为 1 再参与综合评价
double wm = parseW("fce.weights.manager", 0.4);
double ws = parseW("fce.weights.system", 0.35);
double wp = parseW("fce.weights.peer", 0.25);
double sum = wm + ws + wp;
wm /= sum;
ws /= sum;
wp /= sum;

// 每人：上级评价行 mgr、系统客观行 sys、同伴行 peer → 模糊综合得总分
double score = FuzzyComprehensiveEvaluation.score(new double[][]{mgr, sys, peer}, new double[]{wm, ws, wp});
```
