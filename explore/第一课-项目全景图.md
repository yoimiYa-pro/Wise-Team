# 第一课：项目全景图 —— 从零开始认识这个系统

## 一、这个项目是什么？

### 1.1 用大白话说

想象一下，你是一个小团队的领导。你的团队有10个人，每天有很多任务要分配。你会遇到这些问题：

- **问题一**：张三很能干，但已经忙得不可开交；李四比较闲，但能力一般。这个新任务该给谁？
- **问题二**：上周布置的任务，今天一看，已经延期三天了！为什么没人告诉我？
- **问题三**：年底要评绩效了，张三说自己干了很多活，李四也说自己贡献很大。到底谁说得对？

这个项目就是一个**"智能小管家"**，帮你解决这些问题：
- 自动推荐"谁最适合做这个任务"
- 自动预警"这个任务可能要延期"
- 自动计算"每个人的绩效得分"

### 1.2 项目的正式名称

**融合智能决策的中小团队任务管理与绩效评估系统**

别被这个名字吓到，拆开看就很清楚：
- **中小团队**：适合10-50人的小公司或部门
- **任务管理**：管理日常工作任务（谁做、什么时候做完）
- **绩效评估**：客观评价每个人的工作表现
- **智能决策**：用数学算法帮领导做决定

## 二、这个项目怎么工作的？

### 2.1 整体架构图

```
┌─────────────────────────────────────────────────────────────┐
│                        用户（浏览器）                          │
│                    在网页上点击、查看、操作                      │
└─────────────────────────────────────────────────────────────┘
                              ↕ 网络请求（就像打电话）
┌─────────────────────────────────────────────────────────────┐
│                      前端（Vue 3）                            │
│                    网页界面，负责展示和交互                      │
└─────────────────────────────────────────────────────────────┘
                              ↕ API接口（就像服务员传递菜单）
┌─────────────────────────────────────────────────────────────┐
│                      后端（Spring Boot）                      │
│                    处理业务逻辑、计算、存储                      │
└─────────────────────────────────────────────────────────────┘
                              ↕ 数据存取（就像往仓库存东西）
┌─────────────────────────────────────────────────────────────┐
│                      数据库（MySQL）                          │
│                    存储所有数据：用户、任务、绩效等                │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 用餐厅比喻理解

| 概念 | 餐厅比喻 | 在项目中的作用 |
|------|----------|----------------|
| 前端 | 餐厅的大堂和菜单 | 用户看到和操作的界面 |
| 后端 | 餐厅的厨房 | 处理各种业务逻辑 |
| 数据库 | 餐厅的仓库 | 存储所有数据 |
| API接口 | 服务员 | 前后端之间传递信息 |

### 2.3 数据流动过程

以"创建一个新任务"为例：

```
1. 用户在前端填写任务信息（任务名称、截止日期等）
   ↓
2. 前端把信息打包，通过API接口发送给后端
   ↓
3. 后端收到信息，进行各种处理：
   - 检查信息是否完整
   - 计算谁最适合做这个任务
   - 把数据存到数据库
   ↓
4. 后端返回结果给前端
   ↓
5. 前端显示"创建成功"，用户看到新任务
```

## 三、项目中的人是怎么分工的？

### 3.1 三种角色

这个系统有三种用户角色，每种角色能做的事情不同：

```
┌─────────────────────────────────────────────────────────────┐
│  系统管理员 (Admin)                                          │
│  - 管理所有用户账号（创建、禁用、重置密码）                        │
│  - 创建和解散团队                                             │
│  - 查看系统日志                                               │
│  - 设置系统参数                                               │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│  团队管理者 (Manager)                                         │
│  - 创建任务并分配给成员                                        │
│  - 查看团队整体工作负载                                        │
│  - 收到风险预警并处理                                          │
│  - 评定成员绩效                                               │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│  执行成员 (Member)                                            │
│  - 查看分配给自己的任务                                        │
│  - 上报任务进度（完成到30%了、50%了...）                        │
│  - 查看自己的绩效得分                                          │
│  - 给同事打分（匿名互评）                                      │
└─────────────────────────────────────────────────────────────┘
```

### 3.2 为什么要有角色区分？

想象一个公司：
- **管理员**就像公司的HR和IT部门，管账号和系统
- **团队管理者**就像部门经理，管任务和人
- **成员**就像普通员工，执行任务

这样分工可以让每个人只看到和操作自己需要的东西，既安全又高效。

┌──────────────────────────────────────────────────────────────┐

### 4.2 功能之间的关系

这些功能不是孤立的，它们像一个链条一样连接：

```
用户登录 → 创建团队 → 创建任务 → 分配任务 → 执行任务 → 上报进度
    ↑                                              ↓
    └──────────── 风险预警 ←── 监控进度 ←─────────────┘
                        ↓
                    任务完成
                        ↓
                    绩效评估
                        ↓
                    数据看板展示
```

## 五、项目用了什么技术？

### 5.1 后端技术栈

| 技术 | 作用 | 用大白话说 |
|------|------|-----------|
| Java 17 | 编程语言 | 写代码用的语言，就像写文章用中文 |
| Spring Boot | 开发框架 | 一个"工具箱"，帮你快速搭建后端 |
| MyBatis | 数据库框架 | 帮你更方便地操作数据库 |
| MySQL | 数据库 | 存储数据的地方，就像一个超级Excel表格 |
| JWT | 身份验证 | 就像一张"通行证"，证明你是谁 |

### 5.2 前端技术栈

| 技术 | 作用 | 用大白话说 |
|------|------|-----------|
| Vue 3 | 前端框架 | 帮你快速搭建网页界面 |
| TypeScript | 编程语言 | JavaScript的升级版，更安全 |
| Ant Design Vue | UI组件库 | 现成的按钮、表格、弹窗等组件 |
| ECharts | 图表库 | 画各种漂亮的图表 |
| Axios | 网络请求库 | 帮前后端传递数据 |
| Vite | 构建工具 | 把代码打包成可以运行的文件 |

### 5.3 为什么选这些技术？

```
Java + Spring Boot
    ↓
优点：成熟稳定、社区活跃、找工作需求大
适合：企业级应用开发

Vue 3 + TypeScript
    ↓
优点：易学易用、性能好、生态完善
适合：现代Web应用开发
```

## 六、项目的代码长什么样？

### 6.1 后端代码结构

```
backend/
├── src/main/java/com/teampm/
│   ├── TeamPerformanceApplication.java  ← 程序入口（从这里开始运行）
│   │
│   ├── domain/          ← 实体类（数据长什么样）
│   │   ├── User.java        - 用户信息
│   │   ├── Task.java        - 任务信息
│   │   └── Team.java        - 团队信息
│   │
│   ├── mapper/          ← 数据访问层（怎么存取数据）
│   │   ├── UserMapper.java  - 操作用户数据
│   │   └── TaskMapper.java  - 操作任务数据
│   │
│   ├── service/         ← 业务逻辑层（怎么处理业务）
│   │   ├── AuthService.java    - 处理登录注册
│   │   └── TaskService.java    - 处理任务相关
│   │
│   ├── web/             ← 控制器层（接收前端请求）
│   │   ├── AuthController.java - 处理登录相关的请求
│   │   └── TaskController.java - 处理任务相关的请求
│   │
│   ├── security/        ← 安全模块（保护系统安全）
│   │   └── JwtTokenProvider.java - 生成和验证通行证
│   │
│   └── algo/            ← 算法模块（智能决策的核心）
│       ├── AhpCalculator.java       - 计算权重
│       └── RiskModel.java           - 预测风险
│
└── pom.xml              ← 项目配置文件（告诉电脑需要哪些工具）
```

### 6.2 前端代码结构

```
frontend/
├── src/
│   ├── main.ts          ← 程序入口
│   ├── App.vue          ← 根组件（整个应用的容器）
│   │
│   ├── views/           ← 页面（用户看到的不同界面）
│   │   ├── LoginView.vue     - 登录页面
│   │   ├── TasksView.vue     - 任务管理页面
│   │   └── DashboardView.vue - 数据看板页面
│   │
│   ├── components/      ← 组件（可复用的小部件）
│   │   └── SkillPickerRows.vue - 技能选择组件
│   │
│   ├── api/             ← API调用（和后端通信）
│   │   └── client.ts        - 封装好的通信工具
│   │
│   ├── router/          ← 路由（决定显示哪个页面）
│   │   └── index.ts         - 页面跳转规则
│   │
│   └── layouts/         ← 布局（页面的骨架）
│       └── MainLayout.vue   - 主布局
│
└── package.json         ← 项目配置文件
```

### 6.3 理解代码层次

把代码结构想象成一个公司的组织架构：

```
┌─────────────────────────────────────────┐
│  Controller (控制器) = 前台接待            │
│  接收用户请求，交给后面处理                 │
└─────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────┐
│  Service (服务层) = 业务部门               │
│  处理具体的业务逻辑                        │
└─────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────┐
│  Mapper (数据层) = 仓库管理               │
│  负责存取数据                             │
└─────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────┐
│  Database (数据库) = 仓库                 │
│  实际存储数据的地方                        │
└─────────────────────────────────────────┘
```

## 七、怎么开始学习？

### 7.1 学习路线图

```
第1课（你现在在这里）
    ↓ 了解项目整体
第2课：后端入门
    ↓ 理解后端基本结构
第3课：数据库与数据层
    ↓ 理解数据如何存储
第4课：业务逻辑层
    ↓ 理解业务如何处理
第5课：前端入门
    ↓ 理解前端基本结构
第6课：前后端通信
    ↓ 理解数据如何传递
第7课：核心算法
    ↓ 理解智能决策的实现
完成！你可以独立开发功能了
```

### 7.2 学习建议

1. **不要着急**：每节课学完再学下一节，稳扎稳打
2. **动手实践**：边看边在代码里找对应的文件
3. **多问为什么**：理解代码为什么这样写，比记住代码更重要
4. **画图理解**：遇到不懂的地方，画个图帮助理解
