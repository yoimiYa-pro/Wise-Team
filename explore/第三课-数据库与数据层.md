# 第三课：数据库与数据层 —— 数据是怎么存储的

## 一、什么是数据库？

### 1.1 用大白话说

想象一个超大的 Excel 表格：
- 有很多"工作表"（表）
- 每个"工作表"有很多"列"（字段）
- 每一行是一条"记录"（数据）

数据库就是一个**"超级 Excel"**：
- 可以存千万级别的数据
- 支持多人同时操作
- 有严格的规则保证数据正确

### 1.2 为什么需要数据库？

```
如果用文件存储数据：

问题 1：查找困难
- 想找"张三"的信息，要逐行读文件

问题 2：并发冲突
- 两个人同时修改，数据就乱了

问题 3：数据关系
- 任务属于某个团队，团队有多个成员...关系复杂

数据库解决了这些问题！
```

### 1.3 MySQL 是什么？

MySQL 是最流行的关系型数据库之一。关系型的意思是：
- 数据之间有"关系"
- 用"表格"来存储数据
- 用 SQL 语言来操作数据

### 2.2 用户表（users）详解

打开 `V1__init.sql`，看看用户表是怎么设计的：

```sql
CREATE TABLE users (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,    -- 用户ID，自动递增
    username VARCHAR(64) NOT NULL UNIQUE,    -- 用户名，不能重复
    password_hash VARCHAR(255) NOT NULL,     -- 密码（加密后）
    display_name VARCHAR(128),               -- 显示名称
    role VARCHAR(32) NOT NULL DEFAULT 'MEMBER', -- 角色
    skills_json JSON,                        -- 技能（JSON格式）
    base_capacity DECIMAL(8,2) NOT NULL DEFAULT 40, -- 每周额定工时
    avg_performance DECIMAL(6,2) NOT NULL DEFAULT 75, -- 平均绩效分
    delay_history_score DECIMAL(6,4) NOT NULL DEFAULT 0, -- 延期历史分
    status TINYINT NOT NULL DEFAULT 1,       -- 状态（1=正常，0=禁用）
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,    -- 创建时间
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP -- 更新时间
);
```

逐字段解释：

| 字段名 | 类型 | 作用 | 用大白话说 |
|--------|------|------|-----------|
| id | BIGINT | 唯一标识 | 每个人有一个身份证号 |
| username | VARCHAR(64) | 登录名 | 你的账号名 |
| password_hash | VARCHAR(255) | 密码 | 加密后的密码，安全！ |
| display_name | VARCHAR(128) | 显示名 | 别人看到的名字 |
| role | VARCHAR(32) | 角色 | ADMIN/MANAGER/MEMBER |
| skills_json | JSON | 技能列表 | 存储技能数据 |
| base_capacity | DECIMAL(8,2) | 额定工时 | 每周能工作多少小时 |
| avg_performance | DECIMAL(6,2) | 平均绩效 | 历史绩效平均分 |
| status | TINYINT | 状态 | 账号是否正常 |

### 2.3 任务表（tasks）详解

```sql
CREATE TABLE tasks (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    team_id BIGINT NOT NULL,                 -- 所属团队
    creator_id BIGINT NOT NULL,              -- 创建人
    assignee_id BIGINT,                      -- 执行人
    title VARCHAR(255) NOT NULL,             -- 任务标题
    description TEXT,                        -- 任务描述
    difficulty DECIMAL(4,2) NOT NULL DEFAULT 1.0, -- 难度系数
    priority INT NOT NULL DEFAULT 3,         -- 优先级（1-5）
    est_hours DECIMAL(10,2),                 -- 预估工时
    deadline DATE,                           -- 截止日期
    progress INT NOT NULL DEFAULT 0,         -- 当前进度（0-100）
    status VARCHAR(32) NOT NULL DEFAULT 'CREATED', -- 状态
    required_skills_json JSON,               -- 所需技能
    risk_level VARCHAR(16) NOT NULL DEFAULT 'GREEN', -- 风险等级
    delay_probability DECIMAL(8,6),          -- 延期概率
    version INT NOT NULL DEFAULT 0,          -- 版本号（乐观锁）
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
);
```

关键概念解释：

**外键（Foreign Key）**：
```
team_id BIGINT NOT NULL,
...
CONSTRAINT fk_tasks_team FOREIGN KEY (team_id) REFERENCES teams (id)
```

用大白话说：`team_id` 必须是 `teams` 表中存在的 `id`。就像你去住酒店，房间号必须是酒店里真实存在的房间。

**JSON 类型**：
```
required_skills_json JSON
```
存储复杂的数据结构，比如：
```json
["Java", "MySQL", "前端开发"]
```

### 2.4 团队成员表（team_members）详解

```sql
CREATE TABLE team_members (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    team_id BIGINT NOT NULL,
    user_id BIGINT NOT NULL,
    approval_status VARCHAR(32) NOT NULL DEFAULT 'PENDING', -- 审批状态
    joined_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    UNIQUE KEY uk_team_user (team_id, user_id), -- 联合唯一索引
    ...
);
```

这个表是一个"中间表"，用于建立用户和团队的多对多关系：

```
一个用户可以加入多个团队
一个团队可以有多个成员

team_members 表就是这个"桥梁"
```

## 三、实体类（Entity）是什么？

### 3.1 实体类的作用

实体类就是**"代码中的数据库表"**：
- 数据库有 `users` 表
- 代码有 `User` 类
- 它们一一对应

### 3.2 User 实体类解析

打开 `domain/User.java`：

```java
package com.teampm.domain;

import lombok.Data;
import java.math.BigDecimal;
import java.time.Instant;

@Data    // Lombok 注解，自动生成 getter/setter/toString 等
public class User {
    private Long id;              // 对应数据库的 id
    private String username;      // 对应数据库的 username
    private String passwordHash;  // 对应数据库的 password_hash
    private String displayName;   // 对应数据库的 display_name
    private String role;          // 对应数据库的 role
    private String skillsJson;    // 对应数据库的 skills_json
    private BigDecimal baseCapacity;      // 对应数据库的 base_capacity
    private BigDecimal avgPerformance;    // 对应数据库的 avg_performance
    private BigDecimal delayHistoryScore; // 对应数据库的 delay_history_score
    private Integer status;       // 对应数据库的 status
    private Instant createdAt;    // 对应数据库的 created_at
    private Instant updatedAt;    // 对应数据库的 updated_at
}
```

### 3.3 字段类型映射

| 数据库类型 | Java 类型 | 说明 |
|------------|-----------|------|
| BIGINT | Long | 长整数 |
| VARCHAR | String | 字符串 |
| TEXT | String | 长文本 |
| DECIMAL(m,n) | BigDecimal | 精确小数 |
| INT | Integer | 整数 |
| TINYINT | Integer | 小整数 |
| DATE | LocalDate | 日期 |
| TIMESTAMP | Instant | 时间戳 |
| JSON | String | JSON字符串 |

### 3.5 Lombok 是什么？

Lombok 是一个"代码生成器"，用注解帮你写样板代码：

```java
@Data
public class User {
    private Long id;
    private String username;
}

// 等价于：

public class User {
    private Long id;
    private String username;
    
    public Long getId() { return id; }
    public void setId(Long id) { this.id = id; }
    public String getUsername() { return username; }
    public void setUsername(String username) { this.username = username; }
    
    public String toString() { ... }
    public boolean equals(Object o) { ... }
    public int hashCode() { ... }
}
```

常用 Lombok 注解：

| 注解 | 作用 |
|------|------|
| @Data | 生成所有 getter/setter/toString/equals/hashCode |
| @Getter/@Setter | 只生成 getter 或 setter |
| @NoArgsConstructor | 生成无参构造函数 |
| @AllArgsConstructor | 生成全参构造函数 |
| @RequiredArgsConstructor | 生成 final 字段的构造函数 |
| @Builder | 生成建造者模式代码 |

## 四、Mapper 接口是什么？

### 4.1 Mapper 的作用

Mapper 是**"操作数据库的工具"**：
- 你告诉它要做什么（查询、插入、更新、删除）
- 它帮你生成 SQL 语句
- 它帮你执行并返回结果

### 4.2 UserMapper 接口解析

打开 `mapper/UserMapper.java`：

```java
package com.teampm.mapper;

import com.teampm.domain.User;
import org.apache.ibatis.annotations.Mapper;
import org.apache.ibatis.annotations.Param;
import java.util.List;

@Mapper    // 标记这是一个 MyBatis Mapper 接口
public interface UserMapper {
    
    User findById(@Param("id") Long id);
    // 根据ID查询用户，返回一个 User 对象
    
    User findByUsername(@Param("username") String username);
    // 根据用户名查询用户
    
    List<User> findAll();
    // 查询所有用户，返回列表
    
    int insert(User user);
    // 插入新用户，返回影响的行数
    
    int update(User user);
    // 更新用户信息
    
    int updatePerformance(@Param("id") Long id, @Param("avgPerformance") BigDecimal avgPerformance);
    // 更新用户绩效分
    
    int updateDelayHistory(@Param("id") Long id, @Param("score") BigDecimal score);
    // 更新延期历史分
    
    int updatePassword(@Param("id") Long id, @Param("passwordHash") String passwordHash);
    // 更新密码
}
```

### 4.3 注解说明

| 注解 | 作用 |
|------|------|
| @Mapper | 标记这是一个 MyBatis Mapper 接口 |
| @Param | 给参数起名字，方便在 SQL 中引用 |

### 4.4 SQL 语句在哪里写？

有两种方式：

**方式一：注解方式**
```java
@Select("SELECT * FROM users WHERE id = #{id}")
User findById(@Param("id") Long id);
```

**方式二：XML 文件方式**
在 `resources/mapper/UserMapper.xml` 中写 SQL（这个项目用的是注解配合 XML）

## 五、MyBatis 是怎么工作的？

### 5.1 MyBatis 的工作原理

```
┌─────────────────────────────────────────────────────────┐
│  Java 代码                                               │
│  User user = userMapper.findById(1L);                   │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│  MyBatis 接收到调用                                       │
│  "哦，你要查询 id=1 的用户"                               │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│  MyBatis 找到对应的 SQL 语句                              │
│  SELECT * FROM users WHERE id = ?                       │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│  MyBatis 把参数填进去，执行 SQL                           │
│  SELECT * FROM users WHERE id = 1                       │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│  数据库返回结果                                           │
│  {id: 1, username: "admin", ...}                        │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│  MyBatis 把结果转换成 User 对象                           │
│  User{id=1, username="admin", ...}                      │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│  返回给 Java 代码                                         │
│  user = User{id=1, username="admin", ...}               │
└─────────────────────────────────────────────────────────┘
```

## 六、数据库迁移（Flyway）

### 6.1 什么是数据库迁移？

数据库迁移就是**"版本控制数据库结构"**：
- 每次修改表结构，都创建一个迁移文件
- 迁移文件按顺序执行
- 可以追踪数据库的变化历史

### 6.2 迁移文件命名规则

```
V{版本号}__{描述}.sql

例如：
V1__init.sql           -- 第一版：初始化
V2__skill_options.sql  -- 第二版：添加技能选项表
V3__task_status_requests_messages.sql -- 第三版：添加任务状态请求和消息
V4__user_profile_requests.sql -- 第四版：添加用户资料请求
```

### 6.3 迁移执行过程

```
┌─────────────────────────────────────────────────────────┐
│  数据库是空的                                             │
└─────────────────────────────────────────────────────────┘
                          ↓
执行 V1__init.sql
创建所有基础表
                          ↓
┌─────────────────────────────────────────────────────────┐
│  数据库有了基础结构                                       │
│  flyway_schema_history 记录：V1 已执行                    │
└─────────────────────────────────────────────────────────┘
                          ↓
执行 V2__skill_options.sql
添加技能选项表
                          ↓
┌─────────────────────────────────────────────────────────┐
│  数据库有了基础结构 + 技能选项表                           │
│  flyway_schema_history 记录：V1, V2 已执行                │
└─────────────────────────────────────────────────────────┘
                          ↓
... 以此类推
```

### 6.4 为什么要用 Flyway？

```
没有 Flyway 时：

开发人员 A："我改了数据库结构，你们手动执行这个 SQL"
开发人员 B："什么 SQL？我没收到啊"
开发人员 C："我执行了，但报错了..."

结果：数据库结构不一致，程序出错！

有 Flyway 时：

启动应用 → Flyway 自动检查 → 自动执行未执行的迁移文件
所有人都同步！
```

## 七、SQL 基础速查

### 7.1 查询（SELECT）

```sql
-- 查询所有用户
SELECT * FROM users;

-- 查询指定用户
SELECT * FROM users WHERE id = 1;

-- 查询用户名和显示名
SELECT username, display_name FROM users;

-- 按创建时间排序
SELECT * FROM users ORDER BY created_at DESC;

-- 分页查询（前10条）
SELECT * FROM users LIMIT 10 OFFSET 0;
```

### 7.2 插入（INSERT）

```sql
-- 插入新用户
INSERT INTO users (username, password_hash, display_name, role)
VALUES ('test', 'hashed_password', '测试用户', 'MEMBER');
```

### 7.3 更新（UPDATE）

```sql
-- 更新用户信息
UPDATE users SET display_name = '新名字' WHERE id = 1;

-- 更新多个字段
UPDATE users SET display_name = '新名字', role = 'MANAGER' WHERE id = 1;
```

### 7.4 删除（DELETE）

```sql
-- 删除用户
DELETE FROM users WHERE id = 1;
```

### 7.5 聚合函数

```sql
-- 统计用户数量
SELECT COUNT(*) FROM users;

-- 计算平均绩效
SELECT AVG(avg_performance) FROM users;

-- 求和
SELECT SUM(est_hours) FROM tasks WHERE assignee_id = 1;
```

## 八、数据流程图

### 8.1 完整的数据流程

```
前端发起请求
      ↓
Controller 接收请求
      ↓
Service 处理业务
      ↓
Mapper 接口调用
      ↓
MyBatis 生成 SQL
      ↓
MySQL 执行 SQL
      ↓
返回结果集
      ↓
MyBatis 转换为 Java 对象
      ↓
返回给 Service
      ↓
返回给 Controller
      ↓
返回给前端
```

### 8.2 以"查询用户信息"为例

```
1. 用户点击"查看个人信息"
      ↓
2. 前端发送 GET /api/auth/me
      ↓
3. AuthController.me() 方法被调用
      ↓
4. 从 JWT 中获取用户 ID
      ↓
5. 调用 userMapper.findById(id)
      ↓
6. MyBatis 执行：SELECT * FROM users WHERE id = ?
      ↓
7. 数据库返回数据
      ↓
8. MyBatis 封装成 User 对象
      ↓
9. 返回给 Controller
      ↓
10. Controller 返回 JSON 给前端
```
