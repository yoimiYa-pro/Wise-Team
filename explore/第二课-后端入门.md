# 第二课：后端入门 —— Spring Boot 基础

## 一、什么是后端？

### 1.1 用大白话说

想象你在一家餐厅吃饭：

- **前端** = 餐厅的菜单和大堂（你看到的、能操作的）
- **后端** = 餐厅的厨房（在幕后处理一切）

当你在前端点击"登录"按钮时，就像你把菜单递给服务员。后端就像厨房：
1. 接收你的请求（服务员把菜单送到厨房）
2. 处理请求（厨师做菜）
3. 返回结果（服务员把菜端上来）

### 1.2 后端到底做什么？

```
后端的三大职责：

1. 接收请求 ─────→ 监听网络请求，知道用户想干什么
                     （比如：用户要登录、用户要查看任务列表）

2. 处理业务 ─────→ 按照规则处理数据
                     （比如：验证密码对不对、计算任务该分给谁）

3. 返回结果 ─────→ 把处理结果送回去
                     （比如：登录成功、返回任务列表）
```

## 二、Spring Boot 是什么？

### 2.1 什么是框架？

框架就像**"半成品房子"**：
- 盖房子需要打地基、砌墙、装门窗...很麻烦
- 框架就是已经帮你打好了地基、搭好了框架
- 你只需要往里面填内容（写业务代码）

### 2.2 Spring Boot 的特点

Spring Boot 是 Java 世界最流行的 Web 框架，它的特点是：

| 特点 | 用大白话说 |
|------|-----------|
| 开箱即用 | 很多功能默认就配置好了，拿来就能用 |
| 约定优于配置 | 只要按照它的规矩来，不用写太多配置 |
| 内置服务器 | 不用单独安装服务器软件，自己就能跑 |
| 生态丰富 | 想要什么功能，基本都有现成的库可以用 |

### 2.3 Spring Boot 的核心概念

```
Spring Boot 的几个关键概念：

┌─────────────────────────────────────────────────────────┐
│  Bean（豆子）                                             │
│  = Spring 管理的对象                                      │
│  就像餐厅里的员工，被餐厅统一管理                          │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│  IOC 容器（控制反转容器）                                  │
│  = 管理所有 Bean 的地方                                   │
│  就像餐厅的人事部，负责招人、分配工作                      │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│  依赖注入（DI）                                           │
│  = 自动把需要的对象给你                                    │
│  就像厨师需要刀，人事部自动给他送一把                      │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│  注解（Annotation）                                       │
│  = 给代码贴标签                                           │
│  就像给员工贴工牌：这个是厨师、那个是服务员                │
└─────────────────────────────────────────────────────────┘
```

## 三、项目是怎么启动的？

### 3.1 入口文件解析

打开 `backend/src/main/java/com/teampm/TeamPerformanceApplication.java`：

```java
package com.teampm;

import org.mybatis.spring.annotation.MapperScan;
import org.springframework.boot.SpringApplication;
import org.springframework.boot.autoconfigure.SpringBootApplication;
import org.springframework.scheduling.annotation.EnableScheduling;

@SpringBootApplication    // 这个注解是核心！
@MapperScan("com.teampm.mapper")    // 扫描 mapper 文件夹
@EnableScheduling    // 开启定时任务功能
public class TeamPerformanceApplication {

    public static void main(String[] args) {
        SpringApplication.run(TeamPerformanceApplication.class, args);
    }
}
```

逐行解释：

```java
@SpringBootApplication    // 这一行做了很多事：
```

这行注解实际上是三个注解的组合：

| 注解 | 作用 | 用大白话说 |
|------|------|-----------|
| @SpringBootConfiguration | 这是一个配置类 | 告诉 Spring：我是配置文件 |
| @EnableAutoConfiguration | 自动配置 | 自动配置各种功能，省得自己配 |
| @ComponentScan | 扫描组件 | 自动找到所有需要管理的 Bean |

```java
@MapperScan("com.teampm.mapper")
```
这行告诉系统：去 `com.teampm.mapper` 文件夹下找所有操作数据库的代码。

```java
public static void main(String[] args) {
    SpringApplication.run(TeamPerformanceApplication.class, args);
}
```
这是程序入口，就像按下"启动"按钮，整个系统开始运行。

### 3.2 启动过程图解

```
按下启动按钮（运行 main 方法）
        ↓
Spring 开始启动
        ↓
扫描所有带注解的类（@Controller、@Service、@Mapper...）
        ↓
创建所有 Bean（就像招工，把所有人都招进来）
        ↓
自动配置各种功能（数据库连接、安全验证...）
        ↓
启动内置服务器（Tomcat），监听 8080 端口
        ↓
系统就绪，等待接收请求！
```

## 四、一个请求是怎么被处理的？

### 4.1 请求处理流程

以"用户登录"为例，看看一个请求是怎么被处理的：

```
用户在前端点击"登录"
        ↓
发送 HTTP 请求到后端
POST http://localhost:8080/api/auth/login
请求体：{"username": "admin", "password": "123456"}
        ↓
    ┌─────────────────────────────────────────────┐
    │              后端处理流程                     │
    │                                             │
    │  1. SecurityConfig（安全检查）               │
    │     - 这个接口是否需要登录？                  │
    │     - 登录接口不需要，放行                    │
    │                                             │
    │  2. AuthController（接收请求）               │
    │     - 找到对应的方法：login()                │
    │     - 接收请求参数                           │
    │                                             │
    │  3. AuthService（处理业务）                  │
    │     - 验证用户名密码是否正确                  │
    │     - 生成 JWT 令牌                          │
    │                                             │
    │  4. UserMapper（查数据库）                   │
    │     - 查询用户信息                           │
    │                                             │
    └─────────────────────────────────────────────┘
        ↓
返回 HTTP 响应给前端
{
    "token": "eyJhbGciOiJIUzI1NiJ9...",
    "refreshToken": "eyJhbGciOiJIUzI1NiJ9..."
}
        ↓
前端收到响应，登录成功！
```

### 4.2 代码详细解析

#### 第一步：SecurityConfig（安全配置）

打开 `SecurityConfig.java`：

```java
@Configuration    // 这是一个配置类
@EnableMethodSecurity    // 启用方法级别的权限控制
@RequiredArgsConstructor    // 自动生成构造函数
public class SecurityConfig {

    private final JwtAuthenticationFilter jwtAuthenticationFilter;

    // 配置安全规则
    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http
            .csrf(AbstractHttpConfigurer::disable)    // 关闭 CSRF 防护
            .sessionManagement(s -> s.sessionCreationPolicy(SessionCreationPolicy.STATELESS))    // 无状态
            .authorizeHttpRequests(auth -> auth
                .requestMatchers("/api/auth/**").permitAll()    // 登录接口公开
                .requestMatchers("/api/admin/**").hasRole("ADMIN")    // 管理员接口需要 ADMIN 角色
                .anyRequest().authenticated()    // 其他接口需要登录
            )
            .addFilterBefore(jwtAuthenticationFilter, UsernamePasswordAuthenticationFilter.class);
        return http.build();
    }
}
```

用大白话说这段代码在干什么：

```
1. 关闭 CSRF：CSRF 是一种网络攻击，但我们用 JWT 验证，不需要这个
2. 无状态会话：不保存用户状态，每次请求都验证令牌
3. 权限规则：
   - /api/auth/** 路径：所有人都能访问（登录、注册）
   - /api/admin/** 路径：只有 ADMIN 角色能访问
   - 其他路径：需要登录才能访问
4. JWT 过滤器：每个请求都先检查 JWT 令牌
```

#### 第二步：AuthController（控制器）

打开 `AuthController.java`：

```java
@RestController    // 这是一个控制器，返回 JSON 数据
@RequestMapping("/api/auth")    // 这个控制器处理 /api/auth 路径的请求
@RequiredArgsConstructor
public class AuthController {

    private final AuthService authService;    // 注入 AuthService
    private final UserMapper userMapper;    // 注入 UserMapper

    @PostMapping("/login")    // 处理 POST 请求，路径是 /api/auth/login
    public Map<String, Object> login(@RequestBody LoginReq req) {
        // @RequestBody：把请求体中的 JSON 自动转换成 Java 对象
        return authService.login(req.getUsername(), req.getPassword());
    }

    @GetMapping("/me")    // 处理 GET 请求，路径是 /api/auth/me
    public User me(@AuthenticationPrincipal UserPrincipal principal) {
        // @AuthenticationPrincipal：自动注入当前登录用户的信息
        User u = userMapper.findById(principal.getId());
        u.setPasswordHash(null);    // 不返回密码
        return u;
    }

    // 内部类：定义登录请求的数据结构
    @Data
    public static class LoginReq {
        @NotBlank    // 不能为空
        private String username;
        @NotBlank
        private String password;
    }
}
```

关键注解解释：

| 注解 | 作用 | 用大白话说 |
|------|------|-----------|
| @RestController | 标记这是一个控制器 | 我是处理请求的 |
| @RequestMapping | 设置基础路径 | 我处理 /api/auth 开头的请求 |
| @PostMapping | 处理 POST 请求 | 用户提交数据时找我 |
| @GetMapping | 处理 GET 请求 | 用户查询数据时找我 |
| @RequestBody | 接收请求体 | 把 JSON 自动转成对象 |
| @AuthenticationPrincipal | 获取当前用户 | 自动告诉你谁在访问 |

#### 第三步：JwtTokenProvider（令牌生成）

```java
@Component    // 这是一个 Bean，交给 Spring 管理
public class JwtTokenProvider {

    private final SecretKey accessKey;    // 访问令牌的密钥
    private final SecretKey refreshKey;    // 刷新令牌的密钥

    // 创建访问令牌
    public String createAccessToken(Long userId, String username, String role) {
        long min = appProperties.getJwt().getAccessMinutes();    // 有效期（分钟）
        Instant exp = Instant.now().plusSeconds(min * 60);    // 计算过期时间
        
        return Jwts.builder()
                .subject(String.valueOf(userId))    // 存用户 ID
                .claim("username", username)    // 存用户名
                .claim("role", role)    // 存角色
                .claim("type", "access")    // 标记类型
                .issuedAt(Date.from(Instant.now()))    // 签发时间
                .expiration(Date.from(exp))    // 过期时间
                .signWith(accessKey)    // 用密钥签名
                .compact();    // 生成最终字符串
    }
}
```

JWT 令牌就像一张**"电子身份证"**：

```
JWT 令牌结构：

┌─────────────────────────────────────────────────────────┐
│  Header（头部）                                           │
│  告诉你这是什么类型的令牌，用什么算法加密                    │
└─────────────────────────────────────────────────────────┘
                         ↓
┌─────────────────────────────────────────────────────────┐
│  Payload（载荷）                                          │
│  存放实际信息：用户ID、用户名、角色、过期时间等              │
└─────────────────────────────────────────────────────────┘
                         ↓
┌─────────────────────────────────────────────────────────┐
│  Signature（签名）                                        │
│  用密钥生成的签名，防止被篡改                              │
└─────────────────────────────────────────────────────────┘

最终生成一串字符串：
eyJhbGciOiJIUzI1NiJ9.eyJzdWIiOiIxIiwidXNlcm5hbWUiOiJhZG1pbiJ9.xxxxx
```

## 五、后端的分层架构

### 5.1 为什么要分层？

想象一个公司：
- 如果所有人做所有事，会很乱
- 所以分工：前台接待、业务部门、仓库管理

代码也一样，分层的目的是**各司其职**：

```
┌─────────────────────────────────────────────────────────┐
│  Controller 层（控制器层）                                │
│  职责：接收请求、返回响应                                  │
│  就像：公司前台                                           │
│  不做业务逻辑，只负责"接待"和"传递"                        │
└─────────────────────────────────────────────────────────┘
                         ↓
┌─────────────────────────────────────────────────────────┐
│  Service 层（服务层）                                     │
│  职责：处理业务逻辑                                        │
│  就像：业务部门                                           │
│  真正干活的地方：验证、计算、判断                          │
└─────────────────────────────────────────────────────────┘
                         ↓
┌─────────────────────────────────────────────────────────┐
│  Mapper 层（数据访问层）                                  │
│  职责：操作数据库                                          │
│  就像：仓库管理员                                         │
│  只负责存取数据，不关心业务                                │
└─────────────────────────────────────────────────────────┘
                         ↓
┌─────────────────────────────────────────────────────────┐
│  Database（数据库）                                       │
│  真正存储数据的地方                                        │
└─────────────────────────────────────────────────────────┘
```

### 5.2 各层代码示例

#### Controller 层（接收请求）

```java
@RestController
@RequestMapping("/api/tasks")
public class TaskController {

    private final TaskService taskService;    // 注入 Service

    @PostMapping    // 创建任务
    public Task create(@RequestBody Task task) {
        return taskService.create(task);    // 只负责传递，不处理逻辑
    }

    @GetMapping("/{id}")    // 查询任务
    public Task getById(@PathVariable Long id) {
        return taskService.getById(id);
    }
}
```

#### Service 层（处理业务）

```java
@Service
public class TaskService {

    private final TaskMapper taskMapper;    // 注入 Mapper

    public Task create(Task task) {
        // 处理业务逻辑
        task.setStatus("PENDING");
        task.setCreatedAt(LocalDateTime.now());
        
        // 调用 Mapper 存数据
        taskMapper.insert(task);
        return task;
    }

    public Task getById(Long id) {
        return taskMapper.findById(id);
    }
}
```

#### Mapper 层（操作数据库）

```java
@Mapper
public interface TaskMapper {

    @Insert("INSERT INTO tasks (...) VALUES (...)")
    void insert(Task task);

    @Select("SELECT * FROM tasks WHERE id = #{id}")
    Task findById(Long id);
}
```

### 5.3 数据流向图

```
前端发送请求
      ↓
┌─────────────────┐
│  Controller     │  接收请求
│  "收到，我来处理" │
└─────────────────┘
      ↓
┌─────────────────┐
│  Service        │  处理业务
│  "让我算算..."   │
└─────────────────┘
      ↓
┌─────────────────┐
│  Mapper         │  存取数据
│  "我去仓库拿数据" │
└─────────────────┘
      ↓
┌─────────────────┐
│  Database       │  数据库
│  (实际存储)      │
└─────────────────┘
      ↓
原路返回给前端
```

## 六、常用注解速查表

### 6.1 类级别注解

| 注解 | 作用 | 示例 |
|------|------|------|
| @Controller | 标记为控制器 | 处理网页请求 |
| @RestController | 标记为 REST 控制器 | 处理 API 请求，返回 JSON |
| @Service | 标记为服务类 | 处理业务逻辑 |
| @Repository | 标记为数据访问类 | 操作数据库 |
| @Mapper | MyBatis 的 Mapper 接口 | 定义数据库操作 |
| @Configuration | 标记为配置类 | 配置各种设置 |

### 6.2 方法级别注解

| 注解 | 作用 | 示例 |
|------|------|------|
| @GetMapping | 处理 GET 请求 | 查询数据 |
| @PostMapping | 处理 POST 请求 | 创建数据 |
| @PutMapping | 处理 PUT 请求 | 更新数据 |
| @DeleteMapping | 处理 DELETE 请求 | 删除数据 |
| @RequestMapping | 通用请求映射 | 可以指定任意 HTTP 方法 |

### 6.3 参数注解

| 注解 | 作用 | 示例 |
|------|------|------|
| @RequestBody | 接收请求体（JSON） | 登录时的用户名密码 |
| @PathVariable | 接收 URL 路径参数 | /tasks/{id} 中的 id |
| @RequestParam | 接收 URL 查询参数 | /tasks?status=PENDING |
| @AuthenticationPrincipal | 获取当前登录用户 | 获取当前用户信息 |
